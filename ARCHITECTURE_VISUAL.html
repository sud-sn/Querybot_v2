Rate & Token Limit
Hard block at limit. Warning at 80%. Token limit = 0 means uncapped.

2
Webhook Dedup
Bloom filter, 5-min TTL. Prevents Slack/Teams re-delivery double-responses.
core/webhook_dedup.py
3
Clarification Check
Glossary ambiguity scan. If term has multiple interpretations → numbered options sent to user. State saved per-user with TTL.
core/clarification.py
4
DDL Guard
Rejects CREATE / DROP / ALTER / TRUNCATE / INSERT / UPDATE / DELETE. Never reaches LLM.
core/llm.py:is_ddl_attempt
5
Schema Scoping
Portal schema tab selection narrows effective table set. Scopes RAG, SQL generation, and validation.
main.py:624
6
RAG — KB Retrieval
Cosine search over querybot_kb. Returns top-k table/column description chunks filtered by user's allowed tables. Synonym injection from KB "Business Synonyms" sections.
core/knowledge.py + core/vector_store.py
7
Entity Graph Resolution
BFS over entity graph → minimal JOIN skeleton. Date role scoring (+35) disambiguates multi-FK date tables. Cycle detection prevents infinite loops.
core/graph_resolver.py
8
Semantic Plan
Matches question terms to metric registry. Injects approved formula + required tables. Up to 3 plans merged (metric + field-level + runtime), deduplicated.
core/semantic_planner.py + core/semantic_model.py
9
Few-Shot Examples
Dual retrieval: querybot_governed (admin-approved) first, querybot_kb fill. Deduped by question text. Governed examples have highest trust.
core/examples.py:356
10
LLM SQL Generation
Assembles system prompt: schema + JOIN skeleton + metric formulas + few-shot + glossary + clarification context. Calls Claude (default) or OpenAI.
core/llm.py
11
SQL Validation
Structural: FQN format, DDL guard, ACL enforcement. Metric formula: AST scan including CTEs. One repair retry on mismatch.
core/validator.py + core/metric_validator.py
12
Cache / DuckDB Router
Same tables + aggregation pattern as previous result → DuckDB re-query. Avoids Snowflake round-trip for follow-up drill questions.
core/result_cache.py + core/query_router.py
13
Execute on Real DB
Snowflake / Oracle / Azure SQL. Per-query connection. Timeout enforced at connector level.
core/schema.py:run_query
14
Zero-Row RCA
Probes table row counts to distinguish "table empty" vs "filter too narrow". Returns business-language hint, not SQL error.
main.py:_build_zero_row_message
15
Response Builder
Confidence score + anomaly callouts + decision signal. LLM-generated narrative summary (1-2 sentences).
core/response_builder.py + core/answer_formatter.py
16

Computes which chips to show based on result shape: time series, ranking, aggregation, trend direction.
core/chip_eligibility.py

Platform adapter (Zoom/Teams/Slack) or WebSocket JSON stream (portal).
gateway adapters + portal WebSocket

Date Role Disambiguation

Enterprise fact tables have many date FKs all pointing to the same physical DIM_DATE. Without role awareness the LLM picks the wrong FK.

12 Built-in Date Roles

RoleKey synonyms
Invoicebilling, invoiced
Orderplaced, sales order
Deliverydelivered, received
Shipdispatch, shipped, sent
Requestrequested, required by
Duedue date, deadline
Creationcreated, entered
Postingposted, booked, GL date
Effectiveeffective from, valid from
Expiryexpires, expiration, ends
Approvalapproved, authorized
Paymentpaid, remittance

Resolution Pipeline

1
Schema discovery writes Pass-3 join map
Each fact FK → DIM_DATE gets its own entry with role synonyms
2
Virtual entities auto-created
"Invoice Date", "Order Date" → same DIM_DATE table, different FK column
3
BFS scoring: +35 for date role match
"invoice month" → Invoice Date entity wins over generic date entity
4
Only winning FK in JOIN skeleton
LLM cannot guess wrong FK — only the correct one is in the prompt

Quality Scoring

Deterministic 0-100. No LLM. Pure signal-based. File: core/quality_scorer.py

Score Breakdown

SignalMax ptsCondition
SQL validation passed10Structural + ACL check OK
Execution success20DB returned result without error
Non-empty result15row_count > 0
No repair needed15First-pass SQL used
Repair succeeded10One retry fixed it
Repair failed0Could not fix
Metric compliance0–20% formulas using approved template
Entity graph compliance0–10JOIN matched entity graph path
Schema ACL compliance10No out-of-scope tables accessed
Total100

Score Classification

Positive ≥ 85 Eligible for approve without correction
Review 60–84 Worth reviewing — may need SQL correction
Negative < 60 Likely bad SQL — reject or known_failure

Feedback Override

Net negative user feedback (thumbs-down > thumbs-up) forces candidate_type = "negative" regardless of technical score. Admin can still approve — explicit human override.


Admin SQL Correction

Admin submits corrected SQL → score = 85, source = admin_correction, source_boost = +0.10 in Genie engine. Corrected SQL is what gets embedded into Qdrant on approve.

Scoring Formula (exact)

# Cold-start guard: < 10 impressions → source quality only if impressions < 10: score = source_boost # governed=0.10 | learned=0.05 | static=0.00 # Warm-start: behavioral signals + confidence blending else: ctr = clicked / impressions exec_rate = executed / impressions success_rate = successful / impressions dismiss_rate = dismissed / impressions behavioral = (0.30 × ctr + 0.40 × exec_rate + 0.25 × success_rate - 0.05 × dismiss_rate) confidence = min(impressions / 100, 1.0) score = confidence × behavioral + (1 − confidence) × source_boost
